# הפעלת מערכת תקנים — מדריך התקנה והפעלה בייצור

מדריך זה מכסה שני מסלולים: הרצה מקומית **במצב הדגמה** (בלי Firebase בכלל, לניסיון/פיתוח), ופריסה
מלאה **לייצור** מול פרויקט Firebase אמיתי + Vercel — כולל כללי אבטחה לפי הרשאות, יצירת המשתמש
הראשון, ומשתני הסביבה הנדרשים בכל סביבה.

---

## מצב הדגמה (Demo Mode) — הרצה מקומית בלי Firebase בכלל

אם אין `.env.local` (או שהוא חסר/פגום), האפליקציה **לא קורסת** — היא עולה אוטומטית במצב הדגמה:
מסך "התחברות" מדולג לגמרי, נכנסים ישר בתור "משתמש הדגמה" בהרשאת Admin. הנתונים נשמרים
ב-**localStorage של הדפדפן בלבד** — טוב לניסיון מהיר או לפיתוח, אבל לא מתאים לשימוש אמיתי בין כמה
אנשים. ברגע שממלאים `.env.local` עם פרטי פרויקט Firebase אמיתי (ראו בהמשך), מצב ההדגמה נעלם
אוטומטית והאפליקציה עובדת מול Firestore אמיתי — אין מתג נפרד לכבות.

לאיפוס נתוני ההדגמה: פתחו את כלי הפיתוח של הדפדפן (F12) והריצו
`localStorage.removeItem('department-d-positions:mock-db:v2')`, ואז רעננו.

## שלב 1 — יצירת פרויקט Firebase

1. היכנסו ל-https://console.firebase.google.com ולחצו "הוסף פרויקט".
2. **Build → Authentication → Sign-in method** → הפעילו "Email/Password".
3. **Build → Firestore Database** → צרו מסד נתונים, במצב **"production"** (לא test mode — כללי
   האבטחה בקובץ `firestore.rules` הם שכבת ההגנה האמיתית, לא מצב הבדיקה של Firestore).
4. **Project settings → General → Your apps** → הוסיפו אפליקציית Web (סמל `</>`), ותנו לה שם.
   Firebase יציג בלוק קונפיגורציה (`apiKey`, `authDomain` וכו') — אלו הערכים לשלב הבא.

**אין צורך "לזרוע" (seed) את מסד הנתונים עם אוספים ריקים** — ב-Firestore אוסף (collection) נוצר
אוטומטית ברגע שנכתב אליו המסמך הראשון. מסד נתונים ריק לגמרי הוא בדיוק נקודת ההתחלה הנכונה; אין
פעולה נוספת נדרשת מלבד לא להריץ ייבוא/נתוני הדגמה כלשהם מולו.

## שלב 2 — משתני סביבה

### מקומית

העתיקו את `.env.local.example` לקובץ בשם `.env.local` ומלאו את הערכים מ-Firebase:

```
NEXT_PUBLIC_FIREBASE_API_KEY=...
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=...
NEXT_PUBLIC_FIREBASE_PROJECT_ID=...
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=...
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=...
NEXT_PUBLIC_FIREBASE_APP_ID=...
```

כל שישה השדות חובה — אם אחד מהם חסר או ריק, האפליקציה תיפול חזרה למצב הדגמה (זו התנהגות
מכוונת, לא תקלה). קובץ `.env.local` לעולם לא מועלה ל-git (מוחרג ב-`.gitignore`).

### ב-Vercel

אותם שישה משתנים בדיוק, מוגדרים תחת **Project → Settings → Environment Variables** בפרויקט
ב-Vercel (ראו שלב 7). מכיוון שכולם מתחילים ב-`NEXT_PUBLIC_`, הם מוטמעים בקוד הצד-לקוח בזמן
build — **כל שינוי בערכים דורש build מחדש** (redeploy), לא רק שמירה.

## שלב 3 — פריסת כללי אבטחה (Firestore Rules)

```
npm install -g firebase-tools
firebase login
firebase use --add   # בחרו את הפרויקט שיצרתם בשלב 1
firebase deploy --only firestore:rules,firestore:indexes
```

`firestore.rules` בפרויקט מגדיר גישה לפי שלושת התפקידים הקיימים במערכת:

| | צפייה (Viewer) | עורך (Editor) | מנהל מערכת (Admin) |
|---|---|---|---|
| קריאה (כל האוספים) | ✓ | ✓ | ✓ |
| יצירה/עדכון תקנים, עובדים, שיבוצים, יחידות, סעיפי תקציב, שינויים עתידיים | ✗ | ✓ | ✓ |
| יצירת רשומות ביומן השינויים | ✗ | ✓ | ✓ |
| ניהול משתמשים (יצירה/הרשאות/השבתה) | ✗ | ✗ | ✓ |
| שינוי הגדרות מערכת (הסף בבקרה הניהולית) | ✗ | ✗ | ✓ |
| מחיקה של כל רשומה שהיא | ✗ | ✗ | ✗ |

מחיקה חסומה לגמרי לכל התפקידים, בכל האוספים — תואם את עקרון הבסיס של המערכת: שום דבר לא נמחק,
רק משנה סטטוס (לדוגמה: תקן עובר ל"בביטול", לא נעלם). יומן השינויים חסום גם לעדכון/מחיקה אחרי
יצירה — רשומת ביקורת היא Append-only מרגע שנוצרה.

גישה לכל אוסף מותנית ב"משתמש פעיל" — כלומר מסמך תואם ב-`users/{uid}` עם `active: true`. זו
הסיבה שהמשתמש הראשון (שלב 4) חייב להיווצר בדרך אחרת, ולא דרך המסך.

## שלב 4 — יצירת המשתמש הראשון (Admin)

הכללים דורשים מסמך ב-`users/{uid}` כדי שמשתמש יוכל להיכנס בכלל — כלומר המנהל הראשון חייב
להיווצר לפני שיש בכלל מנהל שיכול ליצור אותו דרך מסך "ניהול משתמשים". שתי דרכים:

### אפשרות א' — סקריפט אוטומטי (מומלץ)

1. בקונסולת Firebase: **Project settings → Service accounts → Generate new private key**.
   ישמר קובץ JSON — שמרו אותו בשורש הפרויקט בשם `serviceAccountKey.json` (הקובץ כבר מוחרג
   ב-`.gitignore`, לעולם לא יגיע ל-git).
2. הריצו:
   ```
   npm run bootstrap-admin -- you@example.com "סיסמה-חזקה" "השם שלך"
   ```
   הסקריפט יוצר את משתמש ה-Auth (אם אינו קיים) ואת מסמך ה-`users/{uid}` התואם עם `role: admin,
   active: true` — דרך Admin SDK, שעוקף את כללי האבטחה לחלוטין (בדיוק בשביל המצב הזה).
3. **מחקו את `serviceAccountKey.json` אחרי השימוש**, או לפחות ודאו שהוא לא נשאר בשום מקום
   משותף — מפתח זה מעניק גישת ניהול מלאה לפרויקט כולו.

### אפשרות ב' — ידנית דרך הקונסולה

1. **Authentication → Users → Add user** — צרו את עצמכם עם דוא"ל וסיסמה.
2. **Firestore Database → Start collection** → שם האוסף `users` → מזהה המסמך = ה-UID שהתקבל
   בשלב הקודם → הוסיפו שדות: `email` (string), `displayName` (string), `role` (string) = `admin`,
   `active` (boolean) = `true`, `createdAt` (number) = כל מספר, למשל `1`.

מכאן והלאה, משתמשים נוספים נוצרים במסך "ניהול משתמשים" באפליקציה — אין עוד צורך לגעת בקונסולה.

## שלב 5 — הרצה מקומית מול הפרויקט האמיתי

```
npm install

```

הפס הכתום של "מצב מקומי" אמור להיעלם — סימן שהאפליקציה מדברת עם Firestore האמיתי. התחברו עם
פרטי המנהל שיצרתם בשלב 4.

## שלב 6 — אימות build לייצור

לפני כל פריסה (ובוודאי לפני הפריסה הראשונה):

```
npm run lint
npx tsc --noEmit
npm run build
```

שלושתם אמורים לעבור נקי (0 שגיאות). אם `npm run build` נכשל עם שגיאת רשת סביב גופן Google
("Module not found... internal/font/google") — זו בעיית רשת חולפת בהורדת הגופן, לא שגיאת קוד;
הריצו שוב.

## שלב 7 — פריסה ל-Vercel

1. חברו את מאגר ה-git של הפרויקט לחשבון Vercel (Import Project).
2. Framework Preset: **Next.js** (מזוהה אוטומטית) — אין צורך ב-`vercel.json` או בהגדרות build
   מיוחדות.
3. **Project → Settings → Environment Variables** — הוסיפו את כל שישה משתני
   `NEXT_PUBLIC_FIREBASE_*` מהפרויקט האמיתי (Production **וגם** Preview, אם רוצים שסביבות
   preview יעבדו מול אותו פרויקט Firebase).
4. Deploy. בכל שינוי במשתני הסביבה יש ל-Redeploy (Deployments → ⋯ → Redeploy) — הם מוטמעים
   בזמן build, לא נקראים ב-runtime.
5. הוסיפו את דומיין ה-Vercel (`your-app.vercel.app` וכל דומיין מותאם אישית) ל-**Firebase console
   → Authentication → Settings → Authorized domains**, אחרת התחברות תיכשל משם.

## מגבלות ידועות בגרסה זו

- **אין סנכרון בזמן-אמת בין משתמשים שונים** — נתונים מתעדכנים אוטומטית אחרי פעולה שביצעתם
  בעצמכם (המערכת מרעננת את עצמה), אבל אם שני אנשים עובדים בו-זמנית, כל אחד יראה את השינויים של
  השני רק אחרי ניווט/רענון מצדו. זו החלטת ארכיטקטורה מכוונת (שאילתות חד-פעמיות, לא מאזיני
  `onSnapshot` חיים) ולא תקלה.
- **יומן השינויים גדל ללא הגבלה** — אין מנגנון ארכוב/מחיקה אוטומטי. למסך "יומן שינויים" יש תקרה
  מובנית של 200 הרשומות האחרונות, אך שאר החישובים (מגמות, תובנות) קוראים את היומן המלא; בהיקפי
  שימוש גדולים מאוד לאורך שנים ייתכן שיהיה צורך בפתרון ארכוב — לא רלוונטי בטווח הנראה לעין.
- **ייבוא מאקסל קיים אך במעמד משני** — נשמר לשימוש עתידי, לא הזרימה העיקרית של המערכת (הזנה
  ידנית).

## פתרון תקלות

**"Internal Server Error" בפיתוח מקומי, אחרי מחיקת `.next`** — כמעט תמיד מטמון Turbopack פגום,
כי הפרויקט נמצא בתוך OneDrive וסנכרון הענן מתנגש עם כתיבה שוטפת של Next.js. **פתרון:** עצרו
לגמרי את שרת הפיתוח (Ctrl+C), מחקו את `.next` **רק אחרי שהשרת נעצר**, הריצו `npm run dev` מחדש.
אם קורה שוב ושוב — שקלו להחריג את `.next/` מסנכרון OneDrive.

**`permission-denied` מ-Firestore אחרי התחברות** — כמעט תמיד אחד משלושה: (1) מסמך `users/{uid}`
לא קיים בכלל עבור המשתמש המחובר, (2) קיים אך `active` הוא `false`, (3) קיים אך `role` לא כמצופה.
בדקו ב-Firestore console תחת `users/<ה-uid מ-Authentication>`.

**התחברות נכשלת מ-Vercel אך עובדת מקומית** — כמעט תמיד דומיין ה-Vercel לא נמצא ברשימת
Authorized domains ב-Firebase Authentication (ראו שלב 7.5).

**שינוי במשתני סביבה ב-Vercel לא משפיע** — נדרש Redeploy מפורש; שינוי הערך לבדו לא מפעיל build
חדש.

## יבוא נתונים קיימים מאקסל (זרימה משנית)

לאחר התחברות כ-Admin/Editor: **תפריט → יבוא מאקסל** → העלו את קובץ ה-Excel המקורי (חייב לכלול
גיליון בשם "פירוט תקנים" עם הכותרות בשורה 5). המערכת תציג תצוגה מקדימה עם הערות על שורות
שדורשות בדיקה לפני שמאשרים את הייבוא בפועל.

**חשוב:** לאחר ייבוא יש לעבור על מסך "יחידות ומחלקות" ולהזין את התקן המוקצה לכל סעיף תקציב —
הקובץ המקורי הכיל גרסאות סותרות של הכמות המוקצית, ולכן המערכת לא מנחשת אותה אוטומטית.
